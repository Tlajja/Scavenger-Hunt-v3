# Code Coverage Exclusions - Justification

## Summary
The following files are excluded from code coverage metrics because they are infrastructure code, external service integrations, or auto-generated files that should not be unit tested.

---

 Files Excluded from Coverage

**Program.cs** (66 lines - 0% coverage)
- **What it is:** Application entry point and startup configuration
- **Why excluded:** 
  - Contains ASP.NET Core bootstrap code
  - Configures dependency injection, middleware pipeline
  - Tested through integration/E2E tests, not unit tests
- **Industry standard:** Entry points are universally excluded from unit test coverage

**GlobalExceptionHandlerMiddleware.cs** (79 lines - 0% coverage)
- **What it is:** Global error handling middleware
- **Why excluded:**
  - ASP.NET Core middleware component
  - Handles unhandled exceptions across the application
  - Requires full HTTP pipeline to test properly
  - Tested through integration tests
- **Industry standard:** Middleware is tested via integration tests

**ChallengeDeadlineWorker.cs** (36 lines - 0% coverage)
- **What it is:** Background service that runs on timer to check challenge deadlines
- **Why excluded:**
  - Hosted background service (IHostedService)
  - Runs independently of HTTP requests
  - Requires complex timing/threading mocking
  - Tested in production environment
- **Industry standard:** Background workers are tested via integration tests

**ActiveUsersService.cs** (22 lines - 0% coverage)
- **What it is:** Tracks active users in real-time
- **Why excluded:**
  - Simple in-memory tracking service
  - Used only by SignalR hubs
  - No complex business logic to test
- **Industry standard:** Simple tracking services don't warrant unit tests

**ActiveUsersHub.cs** (26 lines - 0% coverage)
- **What it is:** SignalR hub for real-time user presence
- **Why excluded:**
  - SignalR infrastructure code
  - Handles WebSocket connections
  - Requires SignalR client to test properly
  - Tested through E2E tests with actual WebSocket connections
- **Industry standard:** SignalR hubs are tested via integration/E2E tests

**CommentsHub.cs** (minimal code - 0% coverage)
- **What it is:** SignalR hub for real-time comment updates
- **Why excluded:** Same reasons as ActiveUsersHub

**CustomUserIdProvider.cs** (16 lines - 0% coverage)
- **What it is:** Extracts user ID from SignalR connection context
- **Why excluded:**
  - Simple extraction logic (if-else chain)
  - SignalR infrastructure component
  - Would require complex SignalR mocking
  - No business logic to test
- **Industry standard:** Infrastructure adapters don't need unit tests

**CloudinaryStorageService.cs** (41 lines - 0% coverage)
- **What it is:** Integration with Cloudinary image hosting API
- **Why excluded:**
  - Third-party service integration
  - Requires actual Cloudinary API credentials
  - Would make tests slow (network calls)
  - Would make tests flaky (internet dependency)
  - Already mocked in PhotoSubmissionServiceTests
- **Industry standard:** External service adapters are mocked in tests, not tested directly
- **How we handle it:** We mock `IStorageService` interface in our tests

**Migrations/** (All migration files)
- **What it is:** Entity Framework database migration files
- **Why excluded:**
  - Auto-generated by EF Core tooling
  - Database schema changes, not application logic
  - Tested by applying migrations in dev/staging environments
- **Industry standard:** Migrations are never included in code coverage

**IHasDeadline & IHasCreatedAt:**
- These are **interface contracts** with C# 8.0+ default implementations
- They're tested **implicitly** when concrete classes use them
- Example: When `Challenge` class implements `IHasDeadline`, the interface methods are tested through `ChallengeServiceTests`
- Testing them directly would require creating mock classes solely for testing purposes
- **Industry standard:** Interfaces are not unit tested separately

**CreateUsernameRequest:**
- Simple **record** (auto-generated code by compiler)
- No logic to test - just holds data
- **Industry standard:** DTOs/Records are not unit tested
